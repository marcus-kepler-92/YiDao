# 结构化开发工作流

## 概述

此技能提供一个结构化的开发工作流程，确保在动手写代码之前充分理解上下文和需求，通过分步实现和逐步验证来降低风险，并使用TaskList功能支持工作流的中断和恢复。

**核心原则：**
- 不要靠猜，先读取上下文
- 先讨论方案，再动手实现
- 分步实现，逐步验证
- 一切从简

## 工作流程

### 阶段一：环境感知

在开始任何开发工作之前，必须先了解项目环境：

1. **读取项目规范**
   - 检查并读取 `CLAUDE.md` 或类似的项目说明文件
   - 了解项目的技术栈、代码规范、目录结构

2. **检查可用工具**
   - 检查是否有可用的 MCP 工具
   - 优先使用专用工具而非通用方法

3. **读取相关代码上下文**
   - 根据用户需求定位相关代码文件
   - 理解现有代码结构和模式
   - 不要假设，要实际读取代码

### 阶段二：需求分析

理解上下文后，梳理和分析用户需求：

1. **需求梳理**
   - 用自己的话复述用户需求
   - 列出需求的具体功能点
   - 识别可能的边界情况

2. **合理性分析**
   - 评估需求是否与现有架构兼容
   - 识别潜在的技术难点或风险
   - 如有疑问，向用户确认

3. **任务分解**
   - 使用 TaskList 创建任务列表
   - 将需求拆分为可独立验证的小任务
   - 设置任务之间的依赖关系

### 阶段三：方案讨论

在动手实现之前，与用户讨论技术方案：

1. **提出方案**
   - 描述实现思路
   - 说明会修改哪些文件
   - 解释为什么选择这个方案

2. **获得确认**
   - 等待用户确认方案
   - 如果用户有不同意见，调整方案
   - **未经确认，不要开始实现**

### 阶段四：分步实现

获得确认后，按任务列表逐步实现：

1. **开始任务**
   - 使用 TaskUpdate 将当前任务标记为 `in_progress`
   - 一次只处理一个任务

2. **实现代码**
   - 保持代码简洁，不要过度设计
   - 代码注释使用英文
   - 遵循项目现有的代码风格

3. **验证功能**
   - 运行相关的现有测试
   - 进行手动测试或调用 API 验证
   - 确保没有破坏现有功能

4. **完成任务**
   - 验证通过后，使用 TaskUpdate 将任务标记为 `completed`
   - 继续下一个任务
   - 如果发现新问题，创建新任务

### 阶段五：收尾

所有任务完成后：

1. **确认完成**
   - 检查 TaskList，确保所有任务都已完成
   - 运行完整的测试套件

2. **总结变更**
   - 简要说明做了哪些修改
   - 列出修改的文件
   - 提醒用户可能需要注意的事项

## 中断与恢复

此工作流支持随时中断和恢复：

- **中断时**：TaskList 会保留当前进度状态
- **恢复时**：使用 TaskList 查看进度，从 `in_progress` 或下一个 `pending` 任务继续

## 注意事项

- 沟通使用中文
- 代码注释使用英文
- 不要一次性实现所有功能
- 每完成一步都要验证
- 遇到不确定的情况，先问用户
