## 快速参考

## 单元测试 vs API 测试

### 单元测试 (Unit Tests)
- **位置**: `tests/unit/`
- **测试对象**: Service 层业务逻辑
- **特点**: 
  - ✅ 快速执行（毫秒级）
  - ✅ 隔离测试（使用 Mock）
  - ✅ 不依赖外部资源（数据库、Redis）
  - ✅ 覆盖所有业务逻辑分支

### API 测试 (Integration Tests)
- **位置**: `tests/api/`
- **测试对象**: 完整的 HTTP 请求-响应流程
- **特点**:
  - ⚠️ 较慢（需要启动应用）
  - ⚠️ 测试整个链路（路由、序列化、验证、中间件）
  - ⚠️ 验证 API 契约（请求/响应格式）

## 测试策略建议

### 方案 A：完整测试（推荐）
**单元测试 + API 测试都保留**

- **单元测试**: 覆盖所有业务逻辑、边界情况、错误处理
- **API 测试**: 只保留关键路径（冒烟测试），验证端到端流程

**优点**: 最全面的测试覆盖
**缺点**: 维护成本较高

### 方案 B：以单元测试为主（当前推荐）
**单元测试完整 + API 测试简化**

- **单元测试**: 覆盖所有业务逻辑（已完成 ✅）
- **API 测试**: 只保留核心功能验证
  - ✅ 创建用户（成功/失败）
  - ✅ 获取用户（成功/404）
  - ✅ 获取用户列表（基础分页）
  - ✅ 更新/删除用户（成功）

**优点**: 
- 单元测试快速、全面
- API 测试只验证关键路径
- 维护成本低

**缺点**: API 层面的细节问题可能遗漏

## 建议

**如果单元测试足够完整，API 测试可以简化为：**

1. **保留核心 API 测试**（5-8 个）：
   - POST /api/v1/users - 创建成功
   - GET /api/v1/users/{id} - 获取成功
   - GET /api/v1/users/{id} - 404 错误
   - GET /api/v1/users - 列表查询
   - PUT /api/v1/users/{id} - 更新成功
   - DELETE /api/v1/users/{id} - 删除成功

2. **移除重复的 API 测试**：
   - 查询参数的详细测试（单元测试已覆盖）
   - 边界情况的详细测试（单元测试已覆盖）

3. **保留系统级 API 测试**：
   - /health, /ready 端点
   - /docs, /openapi.json 端点

## 运行测试

```bash
# 运行所有单元测试（快速）
pytest tests/unit/ -v

# 运行所有 API 测试（较慢）
pytest tests/api/ -v

# 运行特定测试类
pytest tests/unit/test_user_service.py::TestUserServiceGet -v

# 运行并查看覆盖率
pytest tests/unit/ --cov=app.services --cov-report=html
```
